B · Executive Summary

What we built, and why it matters.

Timeless Hadith is a fast, beautiful, ad-free reference platform for authentic hadiths of Prophet Muhammad ﷺ. It is built to be searchable in plain English, mobile-first, and trustworthy by design.

What it is

A static-first Islamic reference website (timelesshadith.com) backed by Supabase, with a Next.js admin panel and a Cloudflare Worker that powers AI-grounded search over the hadith corpus.

Why we built it

Most Islamic content on the web is slow, ad-heavy, or hard to verify. We wanted something Apple-clean, instant, and honest — so a parent can hand it to a child without worry.

The problem it solves

Find the right hadith, in the right language, with proper attribution, in under two seconds. Bookmark it. Share it as an image. Read it offline. No clutter.

Where it stands today

Live and stable on Cloudflare Pages. PWA-installable. Admin CMS built. Stripe-backed shop scaffolded but indexing-blocked. Firebase Auth being phased out for Supabase Auth.

✓
Status snapshot. The website is live and serving real users. The 10-phase enhancement sprint (theme system, branding, bookmark badge, featured-hadith carousel, category icons, About page, SEO pass, QA polish, final report) is fully completed as of 2026-03-29.

What lives where

Frontend (root of repo)

Pure HTML pages: index.html, about.html, blog.html, categories.html, category.html, prayer-times.html, privacy.html, terms.html, bookmarks.html, 404.html, offline.html, plus the admin login page admin.html.

Logic in js/: theme-init.js, app.js, supabase-auth.js, supabase-data.js, data.js (static fallback), bookmarks-wire.js, ai-search.js, urdu-toggle.js, page-chrome.js, plus per-page wires.

Single stylesheet: css/styles.css (theme tokens, fonts, components).

Admin app (admin-app/)

Next.js 15 + React 19 with the App Router. Edge runtime everywhere except SEO endpoints (Node.js).

Pages: shop storefront, product detail, cart, checkout, success/cancel, admin dashboard, products, categories, orders, customers, reviews, coupons, settings, analytics, plus a separate SEO dashboard.

API: 23 routes covering shop reads, admin writes (HMAC-protected), Stripe checkout + webhook, and SEO audit/AI/content.

Worker (worker/)

A small Cloudflare Worker at timeless-hadith-worker.ehsaasradio.workers.dev that takes a question, embeds it with OpenAI, runs vector search via Supabase's match_chunks() RPC, and returns a grounded answer with citations.

Edge function (functions/api/)

ai-search.js — a thin proxy from the static site to the Worker. Translates the browser's {query} into the Worker's {question}, applies CORS allow-listing, and rate-limits 10 req/60s per IP.

D · Implementation Methodology

Twelve steps from empty folder to live site.

Each step explains what we did, why it was needed, and a one-line version simple enough for a 12-year-old.

Step 1 · Planning
Decide what the site is — and what it isn't.

What we did: Wrote down the audience (Muslims looking up authentic hadith), the core promise (fast, clean, trustworthy), and the constraints (free hosting, no ads). We picked the domain timelesshadith.com and chose Apple-style minimalism as the design direction.

Why: A clear vision keeps every later decision honest. If a feature doesn't serve the promise, it gets cut.

Kid version: Before drawing the house, we drew the floor plan.

Step 2 · Project Setup
Make a Git repo and a working folder.

What we did: Created the GitHub repo github.com/ehsaasradio-bot/timeless-hadith, added a basic README.md, LICENSE, and .gitignore that excludes .env, node_modules, .next/, .wrangler/, and large archives.

Files changed: .gitignore, README.md, LICENSE, package.json.

Kid version: Got an empty notebook with our name on it.

Step 3 · Frontend Creation
Build the static site by hand, page by page.

What we did: Wrote each public page as a real .html file. Built one shared stylesheet (css/styles.css) using CSS custom properties for the theme tokens (--bg, --ink, --accent). Self-hosted the fonts (Inter, Noto Kufi Arabic, Gulzar) under fonts/ with font-display:swap. Added a service worker (sw.js) and manifest.json to make the site a PWA.

Why: Static HTML on Cloudflare's edge is essentially free and ridiculously fast. No build step means no build errors.

Kid version: Built each room of the house from real bricks instead of pretending with magic.

Step 4 · Backend Integration
Plug Supabase into the front-end for live data.

What we did: Added js/supabase-data.js that initialises the Supabase JS client with the public anon key (read-only by design) and exposes a tiny TH API: TH.categories, TH.getCat(), TH.getHadiths(), TH.findHadith(). Kept js/data.js as a static fallback so the site still works if Supabase is unavailable.

Why: The browser can talk to Supabase directly using the anon key — Postgres' Row-Level Security (RLS) decides what's actually visible. No custom API needed for reads.

Kid version: Connected the website to a giant filing cabinet of hadiths in the cloud.

Step 5 · Database Schema
Design tables for content, shop, and SEO.

What we did: Wrote three migration files in sql/:

  1. 001_schema.sql — documents + document_chunks with vector(1536) embeddings and an HNSW index for semantic search; the match_chunks() RPC for cosine-similarity queries.
  2. 001_shop_schema.sql — shop_products, shop_categories, shop_orders, shop_reviews, shop_inventory, plus a GIN full-text index on title || description.
  3. 003_seo_schema.sql — seo_audits, seo_audit_issues, seo_pages, seo_recommendations.

All migrations are CREATE ... IF NOT EXISTS so they're safe to re-run.

Kid version: Built shelves before putting books on them — and labelled every shelf.

Step 6 · CMS / Admin Work
A typed Next.js admin behind a password.

What we did: Spun up admin-app/ as a separate Next.js 15 project. Built a stateless HMAC-SHA256 session: the browser POSTs the password to /api/admin/login, the server derives a token from HMAC(ADMIN_SECRET, "th-admin-session-v1") and sets it as an httpOnly, secure, sameSite=strict cookie with an 8-hour expiry. Every admin route calls a requireAdmin(request) helper that verifies the cookie with a constant-time comparison.

Why: No external auth dependency. Works on Cloudflare's edge runtime. Easy to reason about.

Kid version: Built a back-office with a key only mum and dad have.

Step 7 · SEO Setup
Title, description, OG, schema, sitemap, robots.

What we did: Wrote unique <title> and <meta name="description"> for every page. Added <link rel="canonical">, Open Graph and Twitter card tags. Embedded JSON-LD blocks (WebSite, Organization, FAQPage) in index.html. Wrote robots.txt and a 102-URL sitemap.xml.

Kid version: Put a clear label on every page so Google can read it without squinting.

Step 8 · Security Hardening
Headers, RLS, env hygiene, admin gates.

What we did: Wrote a _headers file with X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin, Strict-Transport-Security (with preload), and a Content Security Policy that explicitly allow-lists Supabase, the Worker, Aladhan, Open-Meteo, Google Fonts and Cloudflare Insights. Marked admin.html as noindex, no-store. Moved every secret to Cloudflare's encrypted env vars.

Kid version: Added locks, alarms, and a "no kids in the kitchen" sign.

Step 9 · Performance Optimisation
Cache, compress, defer, precache.

What we did: Set Cache-Control: max-age=31536000, immutable for fonts and images, 24h for versioned JS/CSS, 1h for HTML. Generated .webp siblings for every slide JPEG. Registered a service worker (sw.js) that pre-caches the app shell and uses stale-while-revalidate for runtime requests, while bypassing Supabase API calls.

Kid version: Kept the most-used toys at the front of the cupboard so we never have to dig.

Step 10 · Deployment
Connect GitHub to Cloudflare Pages — done forever.

What we did: Created a Cloudflare Pages project pointed at the GitHub repo. Set the build command to nothing (it's a static site) and the output directory to the repo root (via wrangler.toml with pages_build_output_dir = "."). Created a second Pages project for admin-app/ with build command npx @cloudflare/next-on-pages. Deployed the Worker separately with wrangler deploy.

Kid version: Once we hooked it up, every time we save the homework, the teacher sees the new version automatically.

Step 11 · Testing
Real devices, real browsers, real audits.

What we did: Manual QA across iPhone, Android, iPad, desktop Safari, Chrome and Firefox. Ran Lighthouse on every key page. Wrote an SEO audit engine (admin/seo-audit-engine.js) that crawls the live site and stores findings in seo_audits. Verified the offline page works after the SW caches.

Kid version: Tried opening every door of the house, on rainy days and sunny days.

Step 12 · Final Go-Live Checklist
DNS, HTTPS, redirects, monitoring.

What we did: Pointed timelesshadith.com DNS at Cloudflare, enabled "Always Use HTTPS", added a 301 from www. to apex. Confirmed the site loads in < 2s on a 3G profile. Snapshotted a backup bundle (BACKUP-2026-04-12_0948.bundle) before going live.

Kid version: Final walk-through with mum before opening the door to visitors.

F · Security Review

Where we're locked down — and what still needs love.

Authentication

Public site is read-only via Supabase anon key. Admin app uses HMAC-SHA256 stateless sessions stored in httpOnly + secure + sameSite=strict cookies with 8-hour expiry. Low

Supabase RLS

RLS is enabled on documents and document_chunks, but no public read policies have been added yet — reads currently flow through the static js/data.js fallback while the migration finishes. Medium

API key handling

Anon key is correctly used in the browser. Service-role key lives only in Cloudflare's encrypted env. One past leak in scripts/.env required key rotation. High until rotation is verified.

Cloudflare protections

HTTPS-only, HSTS preload, WAF defaults, edge caching. _headers sets X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin. Low

Input validation

Admin API routes do basic String() coercion and trim; no schema validation library yet. Medium

Admin protection

admin.html is noindex, no-store. /shop/admin tests auth on mount and shows a login gate. Every API route calls requireAdmin(). Low

OWASP-style checklist

  • Secrets excluded from git (.env* in .gitignore)
  • Service-role key never in client bundles (rotation required)
  • HTTPS enforced (HSTS preload)
  • Clickjacking protection (X-Frame-Options: DENY)
  • MIME sniffing disabled
  • Strict referrer policy
  • Stripe webhook signature verified
  • Constant-time admin token comparison
  • Content-Security-Policy (allows 'unsafe-inline' scripts — narrow further)
  • RLS policies on documents / document_chunks (enable + write)
  • Schema validation (Zod) on admin POST routes
  • Per-route rate-limiting on admin API
  • Supabase audit-log monitoring
  • Pre-commit secret-scan hook (git-secrets)

Top remaining risks

  1. Service-role key rotation after the past scripts/.env incident.
  2. RLS policies missing on the hadith corpus; finalise the migration off js/data.js.
  3. Admin secret strength — replace any short admin password with 32+ random bytes (openssl rand -base64 32).
  4. CSP unsafe-inline — move toward nonce-based CSP.
  5. Input validation — add Zod schemas to admin API routes.

H · Performance Review

The site is fast. Here's why.

Render-blocking

One stylesheet, one early theme-init script (intentional, prevents flash). All other scripts defer'd. Inline critical hero CSS for the homepage.

Fonts

Self-hosted woff2 with font-display: swap. Inter, Noto Kufi Arabic, Gulzar. Pre-cached in the service worker. Zero external font calls.

Images

Slides shipped as .webp with .jpg fallback. og-image in both formats. Cache header: max-age=31536000, immutable.

Service worker

Precache app shell (HTML, CSS, JS, manifest, fonts, offline page). Stale-while-revalidate for runtime GETs. Bypass Supabase + Cloudflare Insights so APIs stay fresh.

Caching tiers

HTML: 1 hour, must-revalidate. Versioned JS/CSS: 24 hours. Fonts/images: 1 year, immutable. Admin pages: no-store.

Bundle size

Static site: ~0 framework JS. Admin app: 6 production deps (Next, React, Supabase, Stripe, Tailwind, Framer Motion). No bloat.

Top recommendations

  1. Add <picture> with .webp sources to every <img> still using JPEG.
  2. Bump the service-worker CACHE_NAME in lockstep with deploys (avoid stale shells).
  3. Trial AVIF for slide hero (extra 25–35% saving).
  4. Add a preconnect hint to the Supabase REST host on pages that read live data.
  5. Run a Lighthouse CI workflow on every push.
I · UI / UX Review

Apple-clean, deliberately quiet.

Design language

White-first, tight type, generous spacing, soft shadows, no emojis, no italics. Premium feel from the absence of noise.

Theme system

[data-theme="light|dark"] on <html>. Light is the default for first-time visitors. Preference persists in localStorage under th_theme. A migration flag (th_theme_default_v3) one-time-resets legacy forced-dark users.

Responsiveness

Breakpoints at 960px (footer), 768px (nav collapse), 680px. Tested down to 320px. Mobile featured-hadith uses a 40px-threshold horizontal swipe; desktop gets prev/next buttons.

Accessibility

Skip link, ARIA on nav/buttons/pagination, semantic <main>/<nav>/<article>, focus-visible outlines. Gap: no prefers-reduced-motion handling yet.

Bookmarks

Stored client-side under th_bookmarks. The nav shows a rounded pill counter that hides at 0 and caps at 99+.

Kid-friendly explanation

Clean = like a tidy bedroom. You find what you want immediately, nothing yells, and the colours change at night-time so your eyes don't hurt.

K · Deployment Methodology

Push to main, the rest is automatic.

Local development

npx serve .

# Admin app
cd admin-app

# Worker (optional)
cd ../worker

GitHub flow

  1. Branch off main: git checkout -b feat/short-description.
  2. Commit using conventional style: feat(scope): summary, fix(scope): ..., docs: ..., chore: ....
  3. Open a PR. CI workflow .github/workflows/pages.yml validates that the seven core HTML files exist.
  4. Merge to main.

Cloudflare Pages

Two Pages projects are connected to the same repo:

  • Main site — root output dir . (no build step). Auto-deploys on push to main.
  • Admin app — build command npx @cloudflare/next-on-pages, output admin-app/.vercel/output/static.

The Worker is deployed separately with wrangler deploy and lives at timeless-hadith-worker.ehsaasradio.workers.dev.

Environment variables

Set in the Cloudflare dashboard, never in code:

  • Public: NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY, NEXT_PUBLIC_SITE_URL.
  • Server-only: SUPABASE_SERVICE_ROLE_KEY, ADMIN_SECRET, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, RESEND_API_KEY, OPENAI_API_KEY, EMAIL_FROM, ADMIN_EMAIL.

Production checklist

  • Custom domain points at Cloudflare (orange cloud on)
  • "Always Use HTTPS" enabled
  • HSTS preload
  • Apex + www 301 redirect agreed
  • All env vars set in both Pages projects
  • Sitemap submitted to Google Search Console
  • Cloudflare Pages "auto-deploy on push" verified
  • Service worker version bumped

Rollback plan

Cloudflare Pages keeps every deployment. To roll back: open the project in the dashboard, find the previous good deployment, click "Rollback to this deployment". For the database, every meaningful schema change has a corresponding migration file in sql/; reverse migrations should be written before any destructive change ships.

M · Learning Guide for Kids

Build this website like a 12-year-old can understand.

No jargon, no condescension. Just the real ideas, in plain words.

HTML

What it is: the bricks of a webpage. Each tag is a labelled brick. <h1> is a big sign, <p> is a paragraph, <img> is a picture, <a> is a door to another page.

Analogy: Lego pieces. They click together to make a shape.

CSS

What it is: the paint and furniture. CSS picks colour, size, spacing, and where each brick sits. Same Lego shape, different paint job.

Analogy: the outfit your Lego figure is wearing.

JavaScript / TypeScript

What it is: the brain of the page. JavaScript makes things do things — click a button, save a bookmark, fade in an animation. TypeScript is JavaScript that wears a name-tag, so the computer can shout when you spell something wrong.

Analogy: the remote control for your Lego car.

Next.js

What it is: a kit that turns React (a way of writing JavaScript) into a full website with pages, folders, and an API. We used it for the admin panel.

Analogy: a Lego instruction booklet that comes with a battery pack already included.

A database

What it is: a giant filing cabinet. You ask a question, it gives back a row of information. The hadiths live in a database called Postgres.

Analogy: the index card box at the back of a library.

Supabase

What it is: a service that gives you a database, a way to log in, and a place to store files — without you having to set up a server.

Analogy: renting a fully-furnished study, instead of buying chairs.

Cloudflare

What it is: the world's fastest postman. When someone in Tokyo asks for our webpage, Cloudflare hands them a copy from Tokyo, not from London.

Analogy: the same lemonade stand in 200 cities.

GitHub

What it is: a memory book for code. Every time you save, GitHub remembers exactly what changed and who changed it. If you break something, you can rewind.

Analogy: a time machine for your homework.

Deployment

What it is: the moment your website goes from your laptop to the internet, where everyone in the world can see it.

Analogy: moving the school art project from your room to the hallway wall.

How it all flows

You type a web address. Cloudflare gives your browser the HTML page from the closest city. Your browser reads the HTML, paints it with CSS, runs the JavaScript, and the JavaScript asks Supabase for hadiths. Supabase looks them up and sends them back. The browser shows them to you. All under two seconds.

😀
Big idea: a website is just a polite conversation between a browser, a delivery service (Cloudflare), a thinking helper (the server), and a memory box (the database). Each one has a small job and trusts the others.

Glossary

RLS — Row-Level Security: a Postgres feature that decides, on every row, whether the caller is allowed to see/edit it.

HMAC — Hash-based Message Authentication Code: a signature you make with a secret key. We use it to mint admin sessions.

Service worker — a tiny script that lives in the browser and can serve cached pages even when the network is gone.

PWA — Progressive Web App: a website that can be "installed" on a phone home screen.

JSON-LD — a JSON dialect for telling search engines what your page is about.

Edge runtime — code that runs on Cloudflare's globally-distributed network, close to the user.

Embedding — a list of 1,536 numbers that represents the meaning of a piece of text.

HNSW — a graph data-structure that finds the most similar embeddings really fast.

RAG — Retrieval-Augmented Generation: ask the database first, then ask the AI to write an answer using only what was found.

Conventional commit — a commit message that starts with a type and scope, e.g. fix(seo): correct canonical.

Idempotent migration — a SQL script you can run twice and the second run does nothing.

Stale-while-revalidate — serve the cached copy now; quietly fetch a fresh one for next time.
