All issues — ordered by severity

CRITICAL

No middleware.ts — /shop/admin/* routes are publicly reachable at the edge

Anyone who knows the URL can navigate to admin pages in the browser. Individual API routes check auth but the page routes themselves have zero edge-level protection. A brute-force or URL-guessing attack bypasses the login screen entirely.

Domain: Security · File: admin-app/middleware.ts (missing)

CRITICAL

No Content-Security-Policy header on the shop

admin-app/public/_headers has HSTS, X-Frame-Options, and X-Content-Type-Options but is missing a CSP entirely. Any XSS vulnerability in the shop has no mitigation layer — scripts can exfiltrate session cookies and payment data.

Domain: Security · File: admin-app/public/_headers

HIGH

NEXT_PUBLIC_SITE_URL is set to localhost:3000

OG image URLs, canonical href tags, absolute redirects, and email confirmation links will all point to localhost in production. This breaks social sharing, SEO canonicals, and any email-based auth flows.

Domain: Production · Fix in: Cloudflare Pages → Settings → Environment Variables

HIGH

ADMIN_SECRET is a weak, guessable password

"PubThere2019" is a dictionary-attackable credential with a year suffix pattern commonly found in breach databases. The timing-safe login code is solid — but only as strong as the secret it guards.

Domain: Security · Fix in: Cloudflare Pages env + .env.local

HIGH

Stripe keys are still placeholders — all payments will fail

.env.local and Cloudflare environment variables contain placeholder Stripe keys (sk_test_placeholder, pk_test_placeholder). Every checkout, webhook, and refund will return a Stripe authentication error the moment a real customer attempts to pay.

Domain: Shop · Fix in: Cloudflare Pages env + .env.local

HIGH

images: unoptimized: true in next.config.ts

Required for Cloudflare Pages static export but disables Next.js image resizing, lazy loading, and WebP/AVIF conversion. Every product image is served at full original resolution. Core Web Vitals (LCP, CLS) will suffer significantly on product pages.

Domain: Performance · File: admin-app/next.config.ts

MEDIUM

Supabase migration 003 not yet applied

shop_products.category_id is still NOT NULL in the live database. The API route now allows null category_id, but the database constraint will reject the insert and surface a 500 error to the admin creating an uncategorised product.

Domain: Shop · File: admin-app/supabase/migrations/003_nullable_category.sql

MEDIUM

6,967 Urdu SQL chunks ready but not yet applied

chunk_01.sql through chunk_10.sql are generated and in the workspace. The urdu column in the hadiths table is still empty. The accordion UI shows "اردو ترجمہ جلد دستیاب ہوگا" (coming soon) for all 7,277 hadiths until these are applied.

Domain: Content · Files: urdu_chunks/chunk_01.sql → chunk_10.sql

MEDIUM

SUPABASE_SERVICE_ROLE_KEY may be incorrect in Cloudflare Pages

The "Invalid API key" error when creating shop categories was traced to a wrong service role key in Cloudflare Pages env vars. Verify the correct key is set (Supabase → Project Settings → API → service_role) and trigger a redeploy.

Domain: Production · Fix in: Cloudflare Pages → Settings → Environment Variables

Remediation plan — phased

Phase 1 Critical security fixes ~90 min · blocks go-live
1

Create admin-app/middleware.ts

Add a Next.js middleware file that intercepts all requests to /shop/admin/* and redirects unauthenticated users to the login page before the route even renders. Use the existing HMAC session cookie as the auth signal.

// admin-app/middleware.ts
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;
  if (pathname.startsWith('/shop/admin')) {
    const session = request.cookies.get('admin_session');
    if (!session?.value) {
      const loginUrl = new URL('/shop/admin/login', request.url);
      loginUrl.searchParams.set('next', pathname);
      return NextResponse.redirect(loginUrl);
    }
  }
  return NextResponse.next();
}

export const config = {
  matcher: ['/shop/admin/:path*'],
};
2

Add Content-Security-Policy to admin-app/public/_headers

Append a strict CSP to the existing _headers file. Start with a permissive policy that matches your actual script/style sources, then tighten it over time.

/shop/*
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: camera=(), microphone=(), geolocation=()
  Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' https://js.stripe.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; frame-src https://js.stripe.com; connect-src 'self' https://*.supabase.co https://api.stripe.com;
Phase 2 High-severity configuration fixes ~45 min · done in Cloudflare dashboard
3

Fix NEXT_PUBLIC_SITE_URL in Cloudflare Pages

Go to Cloudflare Pages → your project → Settings → Environment Variables. Find NEXT_PUBLIC_SITE_URL and set it to https://timelesshadith.com. Save and trigger a redeploy. This fixes OG tags, canonical URLs, and email auth links.

4

Rotate ADMIN_SECRET to a strong random value

Generate a 32-character random secret: on macOS/Linux run openssl rand -base64 32. Update ADMIN_SECRET in both Cloudflare Pages env vars and your local .env.local. Never commit the new value to git.

5

Add real Stripe keys

Get your keys from the Stripe Dashboard → Developers → API Keys. You need three values: NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY (pk_live_…), STRIPE_SECRET_KEY (sk_live_…), and STRIPE_WEBHOOK_SECRET (whsec_…). Set all three in Cloudflare Pages env vars and local .env.local. For the webhook secret, set up a Cloudflare Pages webhook endpoint in the Stripe Dashboard pointing to https://timelesshadith.com/api/shop/webhook.

6

Fix SUPABASE_SERVICE_ROLE_KEY in Cloudflare Pages

Go to Supabase → Project Settings → API. Copy the service_role key (not the anon key). Update SUPABASE_SERVICE_ROLE_KEY in Cloudflare Pages env vars. Trigger a redeploy. This fixes the "Invalid API key" error when creating shop categories.

Phase 3 Data and database fixes ~30 min · done in Supabase SQL Editor
