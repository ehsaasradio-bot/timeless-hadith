02Top Findings at a Glance

The five issues that, if fixed this sprint, will produce the largest step-change in audit grade.

Theme default regression — dark is hard-coded, CLAUDE.md requires light

Critical

index.html line 2 sets <html lang="en" data-theme="dark"> and js/theme-init.js falls back to 'dark'. CLAUDE.md Phase 1 and the project completion checklist both explicitly require Light Theme as the default for first-time visitors. A one-time migration was shipped to flush returning users back to dark — the opposite of the spec.

Impact
Contradicts brand guidelines; accessibility surface for low-vision users; regression vs Phase 1 sign-off
Action
Change default to 'light' in js/theme-init.js line 22 and on the root <html> tag across all 8 pages. Update the migration key (th_theme_default_v3) to reset stale dark preferences.
Effort
15 minutes

8.6 MB unoptimized popup.mp4 sitting in repo root

Critical

popup.mp4 (8.6 MB) is committed to the repository root. It isn't referenced anywhere in HTML, CSS, or JS — grep confirms zero usage. On every clone, CI build, or Cloudflare Pages deploy this file is shipped as part of the artifact and counts against the Pages bandwidth budget.

Impact
Bandwidth waste, slower deploys, bloated git history
Action
Either delete with git rm popup.mp4 or move to archive/ (gitignored). If the video is genuinely needed, re-encode with H.265 + AV1 fallback and add <video preload="none">.
Effort
5 minutes

26 KB of inline CSS in index.html — half the page weight

High

index.html ships 26,668 bytes of inline <style> — roughly 60% of the 44 KB HTML payload. This bypasses the browser cache (CSS is re-downloaded on every navigation) and inflates Core Web Vitals, particularly Largest Contentful Paint on slow 3G.

Impact
Slower LCP/FCP on repeat visits, cache miss, duplication across pages
Action
Extract hero, AI-search, Topics, and Featured-Hadith blocks into css/home.css. Inline only the first-paint critical CSS (nav + hero shell, ~5 KB).
Effort
2 hours

CSP allows script-src-attr 'unsafe-inline'

High

Both _headers and the inline meta CSP in index.html allow script-src-attr 'unsafe-inline'. This permits inline event handlers such as onclick="…" and is the most common DOM-based XSS amplifier. No such handlers appear in the repo today — it's a standing door that should be closed.

Impact
XSS blast-radius widened; penalizes Lighthouse Best Practices
Action
Remove script-src-attr 'unsafe-inline' from both locations. If a specific inline attribute is needed later, use a nonce or move it to an event listener.
Effort
10 minutes + regression test

34 raw .innerHTML writes — XSS surface

High

Thirty-four innerHTML = assignments across js/app.js, js/categories-wire.js, js/bookmarks-wire.js, and js/ai-search.js. Most concatenate hadith content, user queries, or Supabase response fields into template strings. The ai-search module is properly escaped via _esc(); others rely on the data coming from a trusted source.

Impact
If a future admin or content-injection vector lets malicious HTML into Supabase, every page that renders it is vulnerable
Action
Audit each site. Introduce a shared safeHtml() tagged-template helper and replace all raw template-literal innerHTML with it. Prefer textContent where HTML isn't needed.
Effort
4 hours

05Security

Headers, CSP, CORS, secrets, input handling, auth surface.

Good

  • Strict-Transport-Security (1 year, includeSubDomains, preload)
  • X-Frame-Options: DENY + frame-ancestors 'none'
  • X-Content-Type-Options: nosniff
  • Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy locks geolocation/mic/camera/payment
  • Worker uses allow-list CORS (no wildcard in prod)
  • Worker secrets managed via wrangler secret put
  • Service-role key never leaves the Worker
  • Query length guard (500 chars) on AI endpoint

Needs Work

  • script-src-attr 'unsafe-inline' in CSP
  • Pages Function /api/ai-search uses Access-Control-Allow-Origin: *
  • No rate limiting on /api/ai-search (Worker or Pages)
  • No request signing — anyone can hit the Worker from the allow-list origins
  • Missing Cache-Control: no-store on authenticated responses
  • No Content-Security-Policy-Report-Only endpoint

No rate limit on AI search endpoint

High

Both functions/api/ai-search.js and worker/src/index.js accept unlimited POST traffic. An attacker with a loop can burn through the OpenAI embedding + chat budget in minutes. Query length is capped but request count is not.

Action
Add a Cloudflare Turnstile challenge on the AI search form, OR use the Workers KV-backed rate limiter (60 req/hour per IP). Also set a monthly OpenAI spend cap in the dashboard.

Wildcard CORS on Pages Function

Medium

functions/api/ai-search.js line 35 sets Access-Control-Allow-Origin: '*'. The underlying Worker has a proper allow-list, but the Pages proxy invalidates that defense-in-depth layer by exposing the same endpoint to any origin.

Action
Echo the request Origin only if it's in an allow-list matching the Worker's. Fallback to https://timelesshadith.com.

Supabase anon key hard-coded in client JS

Low (by design)

js/supabase-auth.js and js/supabase-data.js both contain the Supabase anon JWT. This is expected — anon keys are public and protected by Row Level Security. Verify RLS policies are actually enforced on the chunks, hadiths, and bookmarks tables before trusting this. An absent RLS policy here would be a critical exfiltration vector.

06CSS Architecture

Structure, specificity, tokens, dark mode, consistency.

Good

  • Single design-token system via CSS custom properties
  • Dark mode via [data-theme="dark"] attribute
  • Accent colors documented with AA-contrast notes
  • Responsive via clamp() for fluid type
  • Apple-style 16px border-radius standard
  • Only 7 !important declarations

Needs Work

  • 26 KB inline CSS in index.html, 4.5 KB each in other pages
  • No PostCSS / build step — CSS shipped unminified
  • 17 inline style="…" attributes across 5 pages
  • Duplicate spacing/typography tokens between inline and styles.css
  • Dark-mode overrides scattered — no single source of truth
  • Media queries use raw pixel breakpoints instead of tokens

No CSS minification / build pipeline

Medium

css/styles.css is 20 KB uncompressed. A run through Lightning CSS or esbuild would drop it to ~12 KB and inline autoprefixer fallbacks only where needed.

Action
Add a minimal build.js that runs on the Cloudflare Pages build step. Target is <15 KB CSS for first paint.

Inline style attributes on 17 elements

Medium

Sprinkled through index.html (10), categories (2), category (2), bookmarks (2), about (1). These are mostly avatar sizing and fallback visibility toggles. They bypass the design token system and interact with CSP hardening.

Missing prefers-reduced-motion guard

Medium

The new .ai-answer-card uses a fade-up keyframe animation on every answer render. Users who've requested reduced motion in their OS will still see it — a WCAG 2.3.3 violation.

Action
Wrap the animation in @media (prefers-reduced-motion: no-preference) in js/ai-search.js.

08Compliance & Privacy

Good

  • Dedicated privacy.html and terms.html
  • No third-party analytics beyond Cloudflare Insights
  • Cloudflare Insights is privacy-preserving (no cookies)
  • Google SSO uses redirect flow, no password handling
  • Service worker respects same-origin boundaries

Needs Work

  • No cookie consent banner (borderline-required under GDPR/ePrivacy if expanding to EU users)
  • Privacy policy does not enumerate OpenAI as a data processor
  • No data-retention clause on user queries hitting /api/ai-search
  • No DPA link for enterprise users
  • Terms page lacks dispute-resolution jurisdiction

OpenAI not disclosed as sub-processor in privacy policy

Medium

User queries from the search bar are sent to OpenAI via the Worker. The privacy policy needs a "Third-Party Services" section naming OpenAI, the data transferred (the user question, no PII), and a retention note.

09Best Practices & Code Hygiene

Clean Worker architecture

Strength

The worker/src/index.js is well-organized: single responsibility, clear section dividers, validates input, uses retrieval + grounded prompts with explicit anti-hallucination rules, and emits structured JSON with latency metrics.

Conventional commits + CLAUDE.md workflow

Strength

Commit history follows the feat/fix/style/chore convention. The CLAUDE.md guardrails (auto-commit, stop-before-push) produce auditable phase boundaries — unusual and valuable on a solo project.

No automated test suite

Low

Zero unit, integration, or end-to-end tests in the repo. On a static site this is acceptable but the Worker + Pages Function layer deserves a thin smoke test.

Action
Add Vitest + a single E2E script that POSTs to /api/ai-search in CI.

Stale audit reports in reports/

Low

Seven prior audit/implementation HTML reports live under reports/. These are properly disallowed in robots.txt but still ship to production. Move to archive/ or gitignore after sign-off.
