4 items
C-01
Admin CMS Fully Accessible Without Authentication
Security

cms.html returns HTTP 200 to unauthenticated users. The auth gate is a single sessionStorage.getItem('th_cms_auth') === 'true' check — trivially bypassed in browser devtools in seconds. Any visitor can open the full admin panel, create/delete hadiths, and publish content.

Remediation Steps
  • Move CMS behind Supabase Auth — require a valid Supabase session JWT server-side before rendering CMS
  • Add a Cloudflare Pages Function middleware that validates the session token on every /cms request
  • Completely remove the sessionStorage-based gate — it provides zero real protection
  • Enable Cloudflare Access to restrict /cms, /admin, /admin-backup to authenticated users only
Effort: 2–3 hrs
Impact: Critical
Timeline: < 24 hrs
C-02
SHA-256 Admin Password Hash Hardcoded in Public HTML
Security

The line var ADMIN_HASH = 'ba541f87...5452' is shipped inside the public cms.html source. Anyone can view-source, extract this hash, run it through Hashcat or crackstation.net, and recover the plaintext password. SHA-256 without salt is cracked by rainbow tables instantly.

Remediation Steps
  • Delete the ADMIN_HASH variable entirely from cms.html
  • Migrate to Supabase Auth — store admin credentials in Supabase users table with bcrypt hashing
  • Never perform password verification on the client side — all auth checks must happen server-side
  • Rotate the compromised admin password immediately
Effort: 1 hr
Impact: Critical
Timeline: < 24 hrs
C-03
Supabase Anon Key Hardcoded in 4 Source Files + GitHub
Security

The Supabase anon JWT is present verbatim in supabase-data.js, supabase-auth.js, functions/api/ai-search.js, and cms.html. Because the repo is public on GitHub, this key is trivially discoverable via GitHub code search. Anyone with this key can query your entire 7,277-hadith database and exploit any weak RLS policies.

Remediation Steps
  • Immediately regenerate the Supabase anon key in the Supabase dashboard (Project Settings → API)
  • Store the new key in Cloudflare Pages environment variables — never commit it to source code
  • Use a build step or Cloudflare Pages Function to inject the key at runtime
  • Audit and tighten all Supabase RLS policies so the anon key can only read, never write
  • Use git-secrets or GitHub secret scanning to prevent future key commits
Effort: 3–4 hrs
Impact: Critical
Timeline: < 24 hrs
C-04
Sitemap Slug Mismatch — 97 of 102 URLs Broken for Google
SEO

97 category page URLs in sitemap.xml use ?cat=book-1 through ?cat=book-97 IDs. The actual live site serves ?cat=prayer-worship style semantic slugs. Google is submitting the wrong URLs to its index — these pages may be either not indexed at all or generating soft-404s, crushing your organic reach across 97 category pages.

Remediation Steps
  • Export all category slugs from Supabase (SELECT id, slug FROM categories)
  • Regenerate sitemap.xml with correct semantic slugs (?cat=prayer-worship format)
  • Submit the corrected sitemap to Google Search Console for immediate re-crawl
  • Add canonical tags to each category page pointing to the correct slug URL
  • Set up automated sitemap regeneration so slugs stay in sync going forward
Effort: 2 hrs
Impact: Critical
Timeline: < 48 hrs

H-01
10-Year JWT Validity Window on Supabase Token
Security

The anon JWT (iat: 1774958078, exp: 2090534078) is valid until 2036 — a decade. Once rotated (see C-03), new tokens must be issued with a 30–90 day maximum expiry. Long-lived tokens massively increase the window of opportunity after any future exposure.

Remediation Steps
  • Configure Supabase JWT expiry to 30 days maximum in Project Settings → Auth
  • Implement token refresh logic in supabase-auth.js using Supabase's built-in session refresh
  • Rotate anon key immediately (already required by C-03)
Effort: 1 hr
Timeline: Week 1
H-02
Global CORS Wildcard (Access-Control-Allow-Origin: *)
Security

Every route including /cms and /admin returns Access-Control-Allow-Origin: * in _headers. This allows any website on the internet to make cross-origin requests to your admin endpoints, enabling CSRF-style attacks and data exfiltration via malicious third-party pages.

Remediation Steps
  • Update _headers: restrict CORS to Access-Control-Allow-Origin: https://timelesshadith.com for all admin paths
  • Use a more restrictive pattern: /cms and /admin pages should never send CORS headers at all
  • The public API (/api/*) may keep CORS open only if public consumption is intentional
Effort: 30 min
Timeline: Week 1
H-03
Legacy Firebase Auth SDK Still Loading on Every Page
SecurityPerformance

firebase-auth.js still imports Firebase SDK v10.14.1 from gstatic.com CDN on every page load. Firebase has been superseded by Supabase Auth. This dead code adds payload weight, creates an unnecessary external CDN dependency, and widens the attack surface with a redundant auth system.

Remediation Steps
  • Remove firebase-auth.js and delete all Firebase SDK script tags from all HTML pages
  • Audit for any remaining window.firebase or initializeApp references and remove them
  • Confirm Supabase Auth fully handles all sign-in flows before removing Firebase
  • Remove the Firebase project from the Firebase console to prevent orphaned resources
Effort: 2 hrs
Timeline: Week 1
H-04
Missing DMARC Record / SPF Softfail
Security

timelesshadith.com has no DMARC DNS record and SPF uses ~all (softfail) instead of -all (hardfail). This means anyone can send spoofed emails impersonating @timelesshadith.com and most mail servers will accept them — enabling phishing attacks against your users and damaging your domain reputation.

Remediation Steps
  • Add DMARC TXT record: v=DMARC1; p=quarantine; rua=mailto:dmarc@timelesshadith.com
  • Update SPF to use -all (hardfail): v=spf1 include:_spf.google.com -all
  • Optionally add DKIM via Cloudflare Email or your sending provider
Effort: 30 min
Timeline: Week 1
H-05
4 Render-Blocking JavaScript Files — No defer Attribute
Performance

All 4 JavaScript files (app.js, supabase-data.js, supabase-auth.js, firebase-auth.js) load synchronously without defer or async. The browser halts HTML parsing until each script is downloaded and executed — directly harming LCP, FCP, and Time to Interactive scores. This is one of the simplest wins available.

Remediation Steps
  • Add defer attribute to all non-critical script tags across all HTML pages
  • Verify execution order is preserved (defer maintains order, async does not)
  • Remove firebase-auth.js entirely (see H-03) to further reduce script payload
  • Expected Lighthouse improvement: +8–12 points on Performance
Effort: 30 min
Timeline: Week 1
H-06
No WebP / AVIF Images — All 16 Images Served as PNG
PerformanceSEO

Every image on the site is a PNG file. Serving WebP reduces file size by 25–40% with no perceptible quality difference. The largest image is 101KB — as a WebP this would be ~60KB. Google's Page Experience signals reward fast-loading pages; image optimization directly improves LCP.

Remediation Steps
  • Convert all 16 PNG images to WebP using cwebp or Squoosh
  • Use picture element with WebP source and PNG fallback for maximum browser compatibility
  • Enable Cloudflare Polish (automatic WebP conversion) as a zero-code supplement
  • Ensure all images have descriptive alt attributes for SEO and accessibility
Effort: 2–4 hrs
Timeline: Week 2
H-07
JavaScript Not Minified — 79.5KB Uncompressed Bundle
Performance

The full JS bundle is 79.5KB of unminified source code. Minification with Terser or esbuild typically reduces JS by 30–50%, bringing the bundle to ~40–55KB. Combined with defer (H-05) and Firebase removal (H-03), the effective payload reduction could be 60%+.

Remediation Steps
  • Add a GitHub Actions build step: npx terser js/*.js --compress --mangle --output dist/
  • Deploy minified files from dist/ instead of raw js/ directory
  • Configure Cloudflare Pages build command to run minification automatically
Effort: 2 hrs
Timeline: Week 2

M-01
Content Security Policy Uses unsafe-inline
Security

The CSP in _headers includes 'unsafe-inline' for script-src, which undermines XSS protection. This allows injected inline scripts to execute, negating one of CSP's primary purposes.

Remediation Steps
  • Audit all inline scripts and move them to external .js files
  • Replace unsafe-inline with specific 'nonce-RANDOM' or 'sha256-HASH' directives
  • Use CSP evaluator at csp-evaluator.withgoogle.com to validate the updated policy
Effort: 4–6 hrs
Timeline: Month 2
M-02
No Subresource Integrity (SRI) on External Scripts
Security

External scripts (Google Fonts, Firebase CDN) load without integrity attributes. If the CDN is compromised, malicious script updates would load silently into your site with no protection.

Remediation Steps
  • Generate SRI hashes: use srihash.org or openssl dgst -sha384 -binary file.js | openssl base64
  • Add integrity="sha384-..." crossorigin="anonymous" to all external script and link tags
  • Self-host critical third-party scripts where possible
Effort: 2 hrs
Timeline: Month 2
M-03
robots.txt Exposes All Admin Paths as an Attack Map
Security

robots.txt lists /admin.html, /admin-backup.html, /cms.html in Disallow directives. This is a public broadcast of exactly which pages to attack — it's the first thing any automated scanner reads.

Remediation Steps
  • Remove all admin paths from robots.txt — Disallow only hurts SEO for pages you want indexed
  • Real security comes from Cloudflare Access + server-side auth, not robots.txt
  • Rename admin pages to non-obvious paths as a secondary defence layer
Effort: 15 min
Timeline: Month 2
M-04
No Rate Limiting on AI Search API
Security

The /api/ai-search endpoint has no rate limiting. Each request triggers an OpenAI GPT-4o-mini call with no throttle. A single automated script could generate thousands of API calls, draining your OpenAI credits in minutes.

Remediation Steps
  • Enable Cloudflare Rate Limiting: 20 requests/minute per IP on /api/* routes
  • Add Cloudflare Turnstile (free CAPTCHA) to the AI search form for additional protection
  • Set OpenAI usage limits in the OpenAI dashboard as a final safety net
Effort: 1 hr
Timeline: Month 2
M-05
Meta Descriptions Exceed Recommended Length
SEO

About page (170 chars) and Categories page (161 chars) exceed the recommended 150-character limit. Google truncates these in search results, cutting off your message mid-sentence and reducing click-through rate.

Remediation Steps
  • Rewrite About meta description to 120–150 chars, ending with a clear CTA
  • Rewrite Categories meta to 120–150 chars with primary keyword near the front
  • Audit all other pages and ensure meta descriptions are 120–150 chars
Effort: 30 min
Timeline: Month 2
M-06
JS/CSS Cache TTL Too Short (4 Hours)
Performance

JavaScript and CSS assets are cached for only 4 hours. Returning visitors re-download all scripts on every visit after 4 hours. Industry standard for versioned static assets is 1 year (31536000 seconds) with cache-busting filenames on deploy.

Remediation Steps
  • Update _headers: set Cache-Control: public, max-age=31536000, immutable for /js/* and /css/*
  • Add a content hash to filenames (app.abc123.js) so cache busting is automatic on each deploy
  • Cloudflare Pages build step can append hashes automatically
Effort: 1 hr
Timeline: Month 2
M-07
Schema.org Markup Missing on Key Pages
SEO

Bookmarks and 404 pages have no Schema.org markup. Individual hadith pages are missing HowTo or Quote schema. Rich results eligibility requires structured data — without it, Google cannot display enhanced SERP features that increase visibility.

Remediation Steps
  • Add Quote schema (schema.org/Quotation) to individual hadith pages with text, author, source
  • Add BreadcrumbList schema to category and hadith pages for SERP breadcrumbs
  • Add FAQ schema to any page containing Q&A content
  • Validate with Google's Rich Results Test: search.google.com/test/rich-results
Effort: 4 hrs
Timeline: Month 2
M-08
PWA Incomplete — Service Worker Missing
PWA

manifest.json is present with correct display:standalone and icon definitions, but no Service Worker is registered. Without a SW, the site cannot work offline, cannot be installed as a PWA on mobile, and misses Lighthouse's PWA audit points — a missed opportunity for engagement.

Remediation Steps
  • Create sw.js with a Stale-While-Revalidate caching strategy for static assets
  • Register it in index.html: navigator.serviceWorker.register('/sw.js')
  • Cache the hadith JSON data for offline reading
  • Use Workbox (free, from Google) to simplify SW setup significantly
Effort: 3–5 hrs
Timeline: Month 2

L-01
Google Fonts Missing font-display: swap Directive
PerformanceSEO

The Google Fonts @import URL in styles.css doesn't include &display=swap. Without this, the browser uses font-display: auto, which can cause invisible text (FOIT) during font load, hurting CLS and user experience.

Remediation Steps
  • Update Google Fonts URL: append &display=swap to the @import query string
  • Better: self-host Inter font to eliminate the external CDN round-trip entirely
Effort: 5 min
Timeline: Month 3
L-02
Only 1 of 3 Images Uses Lazy Loading
Performance

index.html has 3 images but only 1 uses loading="lazy". The other 2 load eagerly at page start regardless of whether they are visible. Lazy loading below-the-fold images reduces initial page load weight.

Remediation Steps
  • Add loading="lazy" to all images that appear below the viewport fold
  • Keep loading="eager" on the hero/LCP image only so it loads immediately
  • Audit all page templates for missing lazy load attributes
Effort: 15 min
