1  ·  Logo — why the "old logo" appeared and how it's fixed

The root cause was not a wrong file on disk — timelesshadith-logo.png in the repo root is already the new stacked wordmark (1200×390, 78 KB, same file committed in d490637). What was showing for you was a stale cached copy plus a slightly muddy dark-mode CSS filter.

Fixes shipped

  • Appended ?v=2 to every timelesshadith-logo.png reference across all seven public pages (nav, footer, and the JSON-LD logo field on about.html). Browsers and CDNs now treat it as a new URL and bypass the stale cache.
  • Deleted the unused legacy timelesshadith Logo.png (2000×600 horizontal single-line wordmark) — it was sitting in the repo root but not referenced anywhere. It's gone so it can't cause confusion again.
  • Bumped .nav-logo img height from 36px to 40px for better optical weight beside the nav links, and added a subtle hover opacity.
  • Swapped the dark-mode logo filter from invert(1) hue-rotate(180deg) (which produced a tinted result) to brightness(0) invert(1) for a clean pure-white rendering.
  • Upgraded the homepage hero mark from favicon-32.png being upscaled to 56×56 (blurry) to icon-192.png for retina-crisp rendering.

2  ·  Dark theme as the new default

Dark is now the canonical default across the entire site. A one-time localStorage migration (th_theme_default_v2) clears any legacy light choice from the old light-default era so returning visitors land on dark. After that, the toggle is fully respected and persisted — no behaviour change on subsequent visits.

Changes

  • js/theme-init.js: one-time migration then saved || 'dark'.
  • offline.html: flipped data-theme="light" to dark so the offline fallback matches the rest of the site.
  • Every main page's paired <meta name="theme-color"> tags (light #ffffff / dark #0a0a0a) replaced with a single unconditional dark value (#060c1a, matches --bg) plus a <meta name="color-scheme" content="dark light"> hint so mobile browsers render the viewport chrome correctly on first paint.

3  ·  Dark-mode CSS polish pass

Targeted review of spacing, contrast, focus and scrollbar behaviour in dark mode. No light-mode rules were touched.

What shifted

  • Contrast: dark --muted lifted from #7c8fb0 to #9aa9c7 for clearer body text and link contrast on deep navy.
  • Borders: dark --border widened from 0.08 to 0.10 alpha for slightly more definition on cards and dividers.
  • Card elevation: added a 1px inset highlight + deeper drop shadow so glassmorphism surfaces read better at a distance.
  • Rhythm: bumped .featured-section margin (56 → 72 px) and .cards-section padding (48/80 → 56/96 px) so the ambient glow + grid background have room.
  • Scrollbar: custom translucent white scrollbar (webkit + firefox) so the default bright system scrollbar stops flashing against the deep navy canvas.
  • Selection: indigo-tinted text selection highlight in dark mode.
  • Focus: crisper :focus-visible ring (solid accent, 2 px offset) because the default browser outline disappears on deep navy.
  • Chrome: softer 0.06-alpha borders on nav and footer separators so the chrome floats cleanly above the background.
  • Links: default anchor color in dark mode swapped to --accent-3 (periwinkle) for better visibility on privacy/terms pages.
  • Drawer & cookie banner: both retuned for the dark glass aesthetic with matching blur, alpha, and shadow values.

4  ·  P4 backlog — WebP variants

Resolved "Generate WebP variants of brand PNGs for an extra ~30% image payload cut". Measured reduction is 56.8% across five assets, almost double the audit's estimate.

Nav logo, footer logo, and the homepage hero mark are now wrapped in <picture> elements with a <source type="image/webp"> plus the existing <img> as a graceful fallback across all seven public pages. WebP-capable browsers (~97%) get the smaller asset; everyone else transparently falls back to PNG with no JS.

PNG originals are intentionally kept for og:image (social platforms), PWA manifest icons, and favicon links where WebP isn't universally supported downstream.

5  ·  P4 backlog — Per-hadith permalinks & internal link block

Resolved "Add per-hadith permalinks with internal link blocks for SEO depth". Every hadith now has a stable shareable permalink of the form /category.html?cat=<slug>&h=<id>.

Deep-link navigation

  • category-wire.js reads ?h=<id> after filters apply, computes the correct page index, renders that page, then smoothly scrolls to and flashes the target card.
  • Each hadith card now carries id="h-<id>" and data-hadith-id so native browser anchor scrolling also works as a fallback.
  • scroll-margin-top: 90px prevents the fixed nav from covering the landing card.
  • New .hadith-card-flash keyframe pulses an indigo glow around the landing card for 2.2s after deep-link arrival.

Internal link block (the SEO depth win)

  • New <aside id="hadith-index-block"> below pagination on category.html.
  • renderInternalLinks() emits a grid of <a rel="bookmark"> anchors for every hadith in the category, giving crawlers a discoverable route to each permalink.
  • Block is hidden while any filter or search is active so filtered views stay clean.
  • Canonical URL stays at the category level on purpose, so deep-link variants don't create duplicate-content issues.
