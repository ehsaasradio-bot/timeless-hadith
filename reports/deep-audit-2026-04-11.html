Executive Summary

Overall maturity: 6.4 / 10. The codebase is in notably better shape than most static-site + Supabase projects I see. It has clean conventional commits, a working service worker, shipped CSP hardening (P3), skip links, prefers-reduced-motion, and a proper sitemap. The P1–P4 audit follow-through is visible in the git history and pays off.

Biggest strengths

  • Mature security headers at the edge via _headers: HSTS preload, X-Frame-Options: DENY, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy locking down geolocation/mic/camera/payment, and a solid CSP that dropped 'unsafe-inline' from script-src in commit f6b315b.
  • Real PWA surface: service worker with stale-while-revalidate, offline fallback, manifest, proper worker-src in CSP.
  • Good SEO foundations: per-page titles/descriptions, canonical tags, sitemap.xml with 97 book URLs, robots.txt hygiene, JSON-LD WebSite schema, semantic HTML, and the new per-hadith permalink scheme just landed in 6da8c66.
  • Dark-first design system driven by CSS custom properties with a clean light/dark split and genuine accessibility affordances (skip links, reduced-motion, color-scheme hint).
  • Clean git hygiene — conventional commits, one logical concern per commit, separate report artefacts under reports/.

Biggest risks

  • Wildcard CORS + zero rate limiting on the AI endpoint (functions/api/ai-search.js). Any third-party site can call POST /api/ai-search from the browser and burn the project's OPENAI_API_KEY. This is the single highest-impact finding in the report.
  • Architectural indexability ceiling — the entire hadith corpus is rendered client-side from Supabase by js/category-wire.js. Googlebot can render JS but it costs crawl budget and delays indexing by days or weeks. At 7,277 hadiths this meaningfully caps organic growth until there is SSR or static pre-rendering.
  • Service worker cache list drift — sw.js precaches file paths that no longer exist (e.g. js/ai-search.js) and is missing the eight wire/helper modules added in commit a8a8f71. First-visit shell caching is partially broken until the cache version is bumped.
  • CSP divergence between edge header and inline meta. _headers and the <meta http-equiv="Content-Security-Policy"> block in index.html specify different allow-lists. When both are present the intersection applies, which creates surprise breakages on future edits.
  • Repo hygiene — two prebuilt timeless-hadith-*.zip archives (~249 KB combined) and a PDF audit report are committed to the repo root. They inflate clones, can leak intermediate build state, and are Cloudflare-deployed as public assets by default.

Top 5 priorities (next 7 days)

  1. Lock down /api/ai-search — origin allow-list (not *), per-IP rate limit (Cloudflare Turnstile or KV-backed token bucket), request size/word count checks. Before any push of the function to production.
  2. Bump service worker VERSION and rewrite CORE_ASSETS to match the current file tree, including the new .webp variants and the eight *-wire.js modules.
  3. Delete the two committed zips and the PDF report from the repo root, add them to .gitignore, and move the archive to reports/ behind the existing Disallow: /reports/ rule if it needs to stay tracked.
  4. Reconcile CSP — remove the inline meta CSP from every HTML page and keep _headers as the single source of truth. Document that script-src-attr 'unsafe-inline' is a known scoped exception until onclick attributes are refactored.
  5. Enable Supabase RLS audit — confirm RLS is on for the hadiths table and that the anon role has read-only policies only. If it isn't, this becomes the number-one priority.

Findings by Category

Security

SEC-01  ·  Wildcard CORS on /api/ai-search with no rate limit Critical
OWASP A05:2021 Security Misconfiguration · OWASP API10:2023 Unsafe Consumption of APIs · OWASP ASVS V13.2, V11

Evidence (functions/api/ai-search.js:22-41):

function _cors(body, status) {
  return new Response(JSON.stringify(body), {
    status: status || 200,
    headers: {
      'Content-Type':                'application/json',
      'Access-Control-Allow-Origin': '*'
    }
  });
}
export async function onRequestOptions() {
  return new Response(null, {
    status: 204,
    headers: {
      'Access-Control-Allow-Origin':  '*',
      'Access-Control-Allow-Methods': 'POST, OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type'
    }
  });
}

The function accepts cross-origin POSTs from any browser, calls Supabase, then forwards the query to https://api.openai.com/v1/chat/completions using the project's OPENAI_API_KEY (line 115). There is no origin validation, no Turnstile/CAPTCHA, no per-IP rate limit, no daily budget cap, no auth, and no bot check. The only protections are a 500-char query cap and the static system prompt.

Why it matters: A single malicious site, bot farm, or even a casual Stack Overflow answer demonstrating the endpoint can drive unlimited spend against the project's OpenAI billing. gpt-4o-mini is cheap per call but billed at zero rate limit this becomes a real incident within hours. This is a direct financial and availability risk.

Fix (ship before next push):
  1. Replace Access-Control-Allow-Origin: '*' with an explicit allow-list: https://timelesshadith.com and https://timeless-hadith.pages.dev. Reject the rest.
  2. Validate the Origin header on POST as well — browsers send it on cross-origin fetches.
  3. Add a Cloudflare KV-backed token bucket: ~20 queries per IP per hour, ~200 per day. CF Pages Functions can read context.request.headers.get('cf-connecting-ip').
  4. Wrap the endpoint with Cloudflare Turnstile and require a token on POST. This takes 15 minutes and shuts down 99% of bot abuse.
  5. Set a hard daily budget in the OpenAI dashboard as a tripwire (e.g. $5/day while you stabilise).
  6. Cache identical queries in KV for 24h — trivially halves cost and latency.
SEC-02  ·  Dual-source CSP drift between _headers and inline meta High
OWASP A05:2021 Security Misconfiguration

Evidence: _headers line 13 sets

Content-Security-Policy: default-src 'self'; script-src 'self' https://accounts.google.com https://static.cloudflareinsights.com; script-src-attr 'unsafe-inline'; ...

and index.html:35-36 sets a second CSP via <meta http-equiv> with a different allow-list:

script-src 'self' https://accounts.google.com https://fonts.googleapis.com https://static.cloudflareinsights.com;

Note the inline version also includes https://fonts.googleapis.com in script-src, which is dead (Google Fonts serves CSS, not JS) and implies a misunderstanding of which directive fonts fall under.

Why it matters: When both CSPs are present the browser takes their intersection. A future edit to one file will silently break policy on pages served with the other. This is a classic source of "CSP worked yesterday" incidents.

Fix: Keep _headers as the single source of truth. Delete the <meta http-equiv="Content-Security-Policy"> line from every HTML page (index, about, categories, category, bookmarks, privacy, terms). Document in CLAUDE.md that CSP lives in _headers and nowhere else.

SEC-04  ·  Supabase anon JWT hardcoded in two places (assumes RLS is on) Medium
OWASP A02:2021 Cryptographic Failures · OWASP ASVS V1.5, V3.5

Evidence: The JWT anon key appears verbatim at:

  • js/supabase-data.js:31 — exposed to every browser
  • functions/api/ai-search.js:18 — as a fallback when env.SUPABASE_ANON_KEY is unset

Supabase anon keys are designed to be public-safe only when Row-Level Security is configured on every exposed table. I could not verify RLS state from the codebase alone.

Why it matters: If RLS is not on, this anon key allows any client to read, write, or delete rows in any exposed table via the PostgREST API, which would be catastrophic. If RLS is correctly configured, this becomes Informational.

Fix (verify now):
  1. Supabase dashboard → Table editor → hadiths (and every other public table) → confirm "Enable RLS" is ON.
  2. Confirm the policies on each table are read-only for the anon role. Example: CREATE POLICY "anon read" ON public.hadiths FOR SELECT TO anon USING (true);
  3. Remove the hardcoded anon fallback from functions/api/ai-search.js and require env.SUPABASE_ANON_KEY to be set — this decouples the function from the frontend key and lets you rotate either independently.
  4. Publish an internal rotation runbook for the anon key.
SEC-05  ·  script-src-attr 'unsafe-inline' retained for onclick attributes Medium
OWASP A03:2021 Injection (XSS surface)

Evidence: _headers:13, retained consciously in P3 commit f6b315b. The live card templates in js/category-wire.js:161-163 build HTML with embedded onclick="TH_LIKES&&TH_LIKES.toggle(...)" strings and inject via innerHTML.

Why it matters: If an attacker ever lands arbitrary text in a hadith field (via Supabase write, compromised admin, or a future CMS bug), the innerHTML path is an XSS vector that the rest of the CSP cannot block because script-src-attr 'unsafe-inline' is allowed. This is accepted technical debt documented in the P3 report, but it stays Medium until the handlers are refactored.

Fix (1–2 day refactor): Replace every inline onclick in card templates with data-* attributes and one delegated addEventListener('click', ...) on #main-content. That lets you drop script-src-attr 'unsafe-inline' entirely. Example skeleton:

// in category-wire.js after list is rendered:
document.getElementById('main-content').addEventListener('click', function (e) {
  var like = e.target.closest('[data-like-id]');
  if (like) { TH_LIKES.toggle(like.dataset.likeId, like); return; }
  var bm = e.target.closest('[data-bookmark-id]');
  if (bm) { TH_BOOKMARKS.toggle(bm.dataset.bookmarkId, bm); return; }
  var share = e.target.closest('[data-share-id]');
  if (share) { TH_SHARE.showModal(_getH(share.dataset.shareId), share.dataset.catTitle); return; }
});
SEC-06  ·  img-src https: is overly permissive Low
OWASP A05:2021

Evidence: _headers:13 sets img-src 'self' data: https:. Any HTTPS host can ship images, which means a compromised template could exfiltrate data via image-URL side channels (?token=...).

Fix: Restrict to the hosts you actually use: img-src 'self' data: https://*.supabase.co https://lh3.googleusercontent.com (the last is for Google OAuth avatars if you keep that login path).
SEC-07  ·  Missing Cross-Origin-Opener-Policy and Cross-Origin-Resource-Policy Low

Evidence: _headers does not set Cross-Origin-Opener-Policy or Cross-Origin-Resource-Policy. These are 2026-baseline hardening headers that prevent cross-origin window abuse and Spectre-class side channels.

Fix (append to /* block): 
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Resource-Policy: same-origin
Cross-Origin-Embedder-Policy: credentialless
The credentialless variant is the right default for a PWA that loads Google Fonts but not authenticated cross-origin resources.
SEC-08  ·  Admin surface location and auth model unverified Informational

The archived archive/admin-backup.html and archive/cms.html contain a working admin shell that was retired to /archive/. robots.txt disallows /archive/, but Cloudflare Pages still serves those files to anyone who knows the path. If anything in those files still references live Supabase tables or acts on user input, they become an unauthenticated admin panel.

Fix: Either delete archive/admin-backup.html and archive/cms.html outright, or add a Cloudflare Access rule on /archive/* to require authentication. Robots.txt is not a security boundary.

WEB-02  ·  Heavy per-page inline CSS duplicates styles.css responsibilities Medium

Evidence: Each HTML page (index.html, category.html, categories.html) contains hundreds of lines of <style> blocks for hero, cards, pagination, etc. css/styles.css is only 530 lines while the inline blocks total several thousand. This produces:

  • Uncacheable CSS (parse-blocking on every page load)
  • Divergent styling between pages (hero margins on index vs category differ inconsistently)
  • Larger first-paint HTML payloads
  • Hard to audit/refactor, especially for dark-mode variants (which right now live in at least three places)
Fix: Extract the inline <style> blocks into css/pages/<page>.css files and include them with <link rel="stylesheet">. Cloudflare caches these aggressively once added to _headers. Budget 2-4 hours.
WEB-03  ·  manifest.json background color is light, app default is dark Low

Evidence: manifest.json:7-8: "background_color": "#ffffff", "theme_color": "#0a0a0a". After commit 2b1cc15 the app defaults to dark. The PWA splash screen will flash white during launch.

Fix: "background_color": "#060c1a", "theme_color": "#060c1a" to match the dark --bg.
WEB-04  ·  js/theme-init.js is not deferred (correct) but is also blocking Low

The script correctly runs synchronously before paint to prevent FOUC. That's the right call. But it's ~25 lines and could be inlined directly into the <head> of each page (with a nonce from CSP) to save a network round-trip on first paint — the file is too small to justify an HTTP request. Because CSP currently forbids inline script, this would require either (a) a per-request CSP nonce from the edge (CF Pages Function), or (b) accepting it as one file request and moving on. Low priority.

Accessibility (WCAG 2.2 AA)

A11Y-01  ·  Touch targets close to 24 px floor in nav and card actions Medium
WCAG 2.2 §2.5.5 Target Size (Minimum) — AA

Evidence: .theme-toggle is 44×26 px (width ok, height 26 px borderline). .card-action buttons in card templates are ~30×30 px including padding. WCAG 2.2 AA requires 24×24 px minimum for the target itself and recommends 44×44 for primary actions.

The nav theme toggle and card-action icons pass the minimum but fail the recommended threshold that improves motor accessibility. On mobile the per-card Like / Bookmark / Share cluster will be error-prone.

Fix: Raise card-action hit area to 44×44 via a transparent padded hitbox while keeping the icon visually small. Example: 
.card-action { position: relative; }
.card-action::before {
  content: ""; position: absolute; inset: -8px;
  /* invisible hit target — 44x44 around a 28x28 icon */
}
A11Y-02  ·  Arabic content lacks lang="ar" on some containers Medium
WCAG 3.1.2 Language of Parts — AA

Evidence: js/category-wire.js:173 wraps the Arabic text with <p class="arabic-text" lang="ar" dir="rtl"> — good. But js/index-wire.js (featured hadith loop) and js/bookmarks-wire.js card template need a spot-check to confirm the same. If Arabic text ever ships without lang="ar", screen readers will mispronounce it as English phonemes.

Fix: Grep all template-building code for arabic-text and ensure lang="ar" dir="rtl" is always present. Add a unit test for the render.
A11Y-03  ·  Dynamic content updates lack aria-live region Medium
WCAG 4.1.3 Status Messages — AA

When filters or search change the visible hadiths on category.html, the card grid is replaced via innerHTML with no aria-live announcement. Keyboard/screen-reader users have no signal that the list changed.

Fix: The <main id="main-content"> should carry aria-live="polite" or — cleaner — announce the count change via a dedicated <div id="sr-status" class="sr-only" aria-live="polite"> block that is updated with "Showing 1–9 of 127 hadiths, filtered by Sahih" on each re-render.
A11Y-04  ·  Focus visibility on dark mode improved but needs 3:1 contrast verification Low
WCAG 2.4.7 Focus Visible (A) · 2.4.11 Focus Not Obscured (AA, new in 2.2)

Commit ee13448 adds outline: 2px solid var(--accent) on dark-mode focus, which is good. Verify the computed contrast between the accent indigo (#4f72f8) and the deep navy background (#060c1a) meets 3:1 for the focus indicator (§1.4.11 Non-text contrast). Quick check: #4f72f8 vs #060c1a is ~5.1:1 — passes.

Also confirm 2.4.11 Focus Not Obscured: the fixed nav is 64 px tall, and the deep-link feature uses scroll-margin-top: 90px — this is correctly handled. Verify the modal (auth and share) returns focus to the trigger on close.

SEO

SEO-01  ·  Entire hadith corpus is client-rendered from Supabase High

Evidence: category.html ships an empty <main id="main-content"> and populates it via js/category-wire.js after fetching from Supabase. Googlebot does execute JS, but render queue wait-times for new URLs are typically 2–14 days, and large client-rendered pages historically underperform. Bing and non-Google crawlers (DuckDuckGo, Yandex) render even less reliably.

Why it matters: At 7,277 hadiths and 97 books, this is the single biggest organic-growth lever in the codebase. The new internal link block (SEC P4 from 6da8c66) helps discovery, but the landing pages themselves are still nearly empty until JS runs.

Fix (ranked):
  1. Static pre-render at build time (best). Add a Node build step that, at deploy time, pulls all 7,277 hadiths from Supabase and writes one static category/<slug>.html with all cards already in the HTML body. This is 1-2 days of work and gives you SSR-equivalent crawlability without adopting a framework.
  2. Ship a plain-text fallback inside <noscript> on each category that lists every hadith number and source — give crawlers a route even without JS.
  3. Per-hadith static pages at /h/<book>/<number>.html — ultimate SEO depth but requires URL rewrites at the edge via CF Pages Functions.
SEO-02  ·  Sitemap covers category roots only; no per-hadith URLs Medium

Evidence: sitemap.xml lists 5 core pages + 97 category books = 102 URLs. The 7,277 hadiths are not in the sitemap. With the new permalink scheme (?cat=X&h=Y) shipped in 6da8c66, every hadith now has a stable URL that Google can index.

Fix: Generate sitemap-hadiths-1.xml, sitemap-hadiths-2.xml, ... as needed (50,000 URL cap per file) and reference them from a sitemap index: 
<sitemapindex xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <sitemap><loc>https://timelesshadith.com/sitemap.xml</loc></sitemap>
  <sitemap><loc>https://timelesshadith.com/sitemap-hadiths.xml</loc></sitemap>
</sitemapindex>
Generate the hadith sitemap at the same build step as the pre-rendered pages.
SEO-03  ·  Sitemap <lastmod> values are static and stale Medium

Evidence: Every URL in sitemap.xml shows <lastmod>2026-04-03</lastmod> or 2026-04-04. These are hand-set strings that will never update automatically. Google uses lastmod as a recrawl hint.

Fix: Generate sitemap.xml at build time from git log --format=%ad on the underlying file (or the Supabase updated_at column for dynamic content).
SEO-04  ·  Thin structured data — only WebSite + SearchAction Medium

Evidence: index.html:38-51. about.html has Organization schema. There is no BreadcrumbList, no Article, no CreativeWork, no Book/Chapter for the 97 books, and no Quotation schema for hadiths themselves.

Fix: On each category page emit a BreadcrumbList (Home → Categories → Book). On individual hadith deep-link loads emit Quotation with spokenByCharacter = narrator and isPartOf = the book as a Book. Google rewards this with rich results eligibility.
SEO-05  ·  URLs use query strings instead of path segments Low

Evidence: /category.html?cat=book-1 rather than /books/sahih-bukhari/revelation/. Google handles query strings fine, but path-style URLs:

  • rank marginally better for keywords in the path;
  • are human-memorable;
  • look cleaner in social shares.
Fix: Add Cloudflare Pages rewrites (_redirects file) to accept /books/<slug> and internally serve /category.html?cat=<slug>. Update the sitemap to emit the new paths.

Performance (Core Web Vitals)

PERF-01  ·  LCP is likely the hero mark image, but Inter font swap can steal the crown Medium

Evidence: The homepage hero mark (56×56 icon-192.png?v=2) is rendered above the fold. Inter font is loaded from fonts.googleapis.com via <link> with a preconnect hint. With display=swap, the Inter font file arrives around 150-400 ms in and the H1 text re-rasters, causing a second LCP candidate.

Fix:
  1. <link rel="preload" as="font" type="font/woff2" href="https://fonts.gstatic.com/s/inter/v..." crossorigin> the specific Inter weight used in the H1 (probably 700 or 800).
  2. Self-host the subset you actually use — 2026 best practice. Google Fonts has been superseded for LCP-sensitive sites by self-hosting via Fontsource or similar, saving a DNS+TLS handshake.
  3. Explicit width/height on the hero mark is already correct (56 × 56) — good, prevents CLS.
PERF-02  ·  INP risk on category page — re-render rebuilds entire card grid on every keystroke Medium

Evidence: js/category-wire.js:131-135 wires the search input directly to applyFilters() which synchronously filters the full array, calls renderPage(), which innerHTML-rebuilds all 9 cards. On a slow phone with 500+ hadiths in memory this pushes INP above the 200 ms "good" threshold.

Fix:
  1. Debounce search input by 120-160 ms (not 0).
  2. Virtualise the card list if it exceeds 50 visible items (you already paginate to 9, so minor).
  3. Replace innerHTML rebuild with DOM diffing — or simply .textContent-update the cards that are still visible between filters and only create/destroy the deltas. Biggest win.
  4. Use scheduler.postTask('user-visible') or requestIdleCallback to split the render.
PERF-03  ·  PNG fallbacks for decorative icons still ship (post-WebP) Low

After commit 79c1948, .png and .webp both exist for the brand assets. The <picture> tag correctly serves WebP to modern browsers. The PNG fallbacks remain in the deploy artifact (needed for legacy + PWA manifest) which is fine, but you can drop bandwidth by another ~15% by serving AVIF as a second <source type="image/avif"> — 2026 browsers all support it.

PERF-04  ·  Unused JS shipped on pages that don't need it Low

Evidence: sw-register.js and page-chrome.js are included on pages where the relevant DOM IDs don't exist; the scripts short-circuit safely but still parse/execute. supabase-data.js (reads 7,277 hadiths) loads on privacy.html and terms.html for no reason.

Fix: Audit each HTML page's <script> block and only include the modules it actually uses. Create a per-page manifest of required scripts. Expected JS savings on privacy/terms: ~60 KB gzipped.

Tech Stack

STACK-01  ·  No package.json / no dependency lockfile / no automated audit Medium

Evidence: There is no package.json, no package-lock.json, no dependency manifest anywhere. All third-party code is vendored or loaded from CDNs directly in HTML. GitHub Actions (.github/workflows/pages.yml) only test -f checks for core files — no linting, no audit, no type check, no Lighthouse CI.

Why it matters: You have no automated visibility when a CDN-loaded dependency (Google Fonts CSS, Cloudflare Insights) changes behaviour or goes away, and no upgrade path for the day you inevitably adopt a build step. You also can't run npm audit, eslint, or prettier.

Fix: Introduce a minimal package.json with dev-only dependencies: eslint, html-validate, stylelint, @lhci/cli (Lighthouse CI). Wire them into the existing GitHub Actions workflow as blocking checks. This is ~3 hours of work and converts your CI from "tests file existence" to "enforces a real standard".
STACK-02  ·  Static HTML + Supabase is the right architecture for this scope Low

The current stack — static HTML on Cloudflare Pages + Supabase PostgREST + one CF Pages Function for AI — is correctly scoped for a hadith reference site. Don't over-react and migrate to Next.js or Astro; the maintainability and cost savings of vanilla HTML are real and match the CLAUDE.md goal of "simple to build and maintain, limited technical skill required".

The one caveat is that the absence of a build step becomes a ceiling once pre-rendering (SEO-01) becomes urgent. At that point the right move is not a framework — it's a 100-line Node script that reads Supabase and writes static HTML. Keep the runtime vanilla.

STACK-03  ·  No observability — no error reporting, no real-user monitoring Low

Tech Stack Recommendation

Verdict: Optimize the current stack. Do not re-platform.

Static HTML + Cloudflare Pages + Supabase + a single Pages Function is the right architecture for a hadith reference site of this scope. It costs near-zero to run, has no framework upgrade risk, and plays to the stated goal of "simple to build and maintain, limited technical skill required" (CLAUDE.md §5).

The two things you actually need to add are:

  1. A build step (Node script, no framework) that runs before each deploy and pre-renders the category pages + generates sitemaps from Supabase. This resolves SEO-01 and SEO-02 simultaneously and is the highest-ROI change in this report.
  2. A package.json + lint/audit CI (STACK-01, OPS-01) so the codebase can grow without quality drift.

Framework options ranked for if/when you outgrow vanilla:

  • Astro — best fit. Keeps your HTML mental model, produces static output, has first-class MDX, and lets you drop in React/Vue islands if needed.
  • 11ty (Eleventy) — even simpler than Astro, no JS runtime at all. Perfect if you stay content-heavy.
  • Next.js — over-powered for this, pulls in React runtime, significantly harder to maintain. Skip unless you adopt a dashboard/admin product.
